# Zarco CRM — full-code audit, 2026-07-07

Three parallel reviews (server actions · UI surface · MCP/OAuth/infra) plus a
schema-level gap analysis against baseline HubSpot/Salesforce use cases.
Everything below was verified against the code, not assumed. Status reflects
the same-day fix session (commits `4fd5a99..e89573a`).

> **Deployment posture: internal-only.** This CRM is for Zarco's own use, not
> sold to customers. There are no external tenant workspaces — only the real
> "Zarco" workspace + a "Demo" one. So the tenant-isolation findings below
> (S0–S3) are **defense-in-depth / data-hygiene, not breach risk**: worst case
> is real and demo data bleeding together, which the fixes also stop. They
> were still worth fixing, but re-read their severity through this lens.
> The one finding whose severity is **unchanged** by internal-only use is the
> email-suppression gap (§4): campaigns and quotes go to real external
> recipients, so UK GDPR/PECR still applies before Resend goes live.

## 1. Security

| # | Finding | Status |
|---|---------|--------|
| S0 | **MCP tools had no workspace scoping** — any valid bearer token could read/edit/delete rows in ANY workspace (RLS is bypassed by the Drizzle role; app-layer filtering is the only tenant boundary). | ✅ FIXED (`4fd5a99`) — all 32 handlers scoped, FK refs validated via `entityInWorkspace`, independently re-audited as complete |
| S1 | **OAuth client IDOR** — `revokeOAuthClient` deleted ANY client by id (cascading all users' tokens); settings listed every client globally. | ✅ FIXED (`a42a303`) — revoke = revoke *my* tokens for that client (finally writes `revokedAt`); listing scoped to clients the caller authorized |
| S2 | **Web-action FK injection** — create/update actions accepted cross-tenant `organizationId`/`dealId`/`contactId`/`subjectId` unvalidated (same class as S0, web side). | ✅ FIXED (`a42a303`) — deals, projects, quotes, quick-create, manual activities, inbox triage, campaigns target-org all validate in-workspace |
| S3 | **Quote public-token gating** — public PDF served drafts; accept/decline worked on never-sent drafts; `validUntil` never enforced; internal PDF route fetched any workspace's quote by id (auth'd but unscoped). | ✅ FIXED (`a42a303`) — drafts invisible on all public paths, expiry enforced (auto-flips to `expired`), id-lookups require workspaceId in the signature |
| S4 | **90-day bearer tokens with no revocation path** — nothing wrote `revokedAt`, no `/oauth/revoke` endpoint, no refresh tokens. | 🔶 PARTIAL — the settings revoke now writes `revokedAt`; a proper RFC 7009 revoke endpoint + refresh tokens still to do |
| S5 | **MCP high-stakes `confirm=true` is model-enforced** — a prompt-injected model can self-confirm deletes/sends. | 🔶 OPEN — move to MCP elicitation when client support lands |
| S6 | **Audit gaps** — MCP deletes + all reads unaudited; public quote actions logged nothing; web actions unaudited. | 🔶 PARTIAL — public quote view/accept/decline now log timeline activities; a durable security log (esp. for deletes/reads) still to do |
| S7 | **Open dynamic client registration** — unlimited anonymous client creation, no rate limiting on OAuth endpoints, consent screen renders attacker-controlled `client_name`. | 🔶 OPEN — consent screen is the only gate; add rate limiting + client-name warning |
| S8 | `send_quote` claimed "no email leaves the system" but flipping status to `sent` publishes the public quote link. | ✅ FIXED — description now says so explicitly; web `markQuoteSent` documents the same |

## 5. Infra notes (unchanged this session)

- **MCP session store on Vercel is in-memory** — `mcp-handler` wants
  `REDIS_URL`/`KV_URL`; neither is referenced in code. Streaming/session-bound
  MCP interactions won't survive across lambda instances. 🔶
- **DB pool**: `max: 10` per serverless instance, no
  `idle_timeout`/`max_lifetime`/`statement_timeout` — the same Supabase-pooler
  failure class that caused the Console chat-hang incident. Recommend
  `max: 1–3` + timeouts. 🔶
- OAuth internals are otherwise solid: PKCE S256 mandatory, constant-time
  verification, tokens stored as SHA-256 hashes, redirect URIs exact-match,
  open-redirect paths validated.
- `seedDemoWorkspace` has no `type='demo'` assertion — only its caller
  guards it. 🔶
